# PocketQuant — 최적화 설계 노트

> PocketQuant의 "전략 탐색"을 최적화 문제로 정식화한 문서.
> 게임 컨셉(포켓몬/체육관) 뒤에 실제로 어떤 최적화가 돌고 있는지 정리한다.

---

## 1. 문제 정의 (현재 — v0.2)

```
maximize   Y(X)
   over    X
```

| 기호 | 의미 | 현재 형태 |
|------|------|----------|
| **X** | 결정변수 = 전략 (어떤 시그널을 쓸지) | 5비트 on/off 벡터 `{0,1}⁵` |
| **Y** | 목적함수 = 적합도(fitness) | 전 체육관 **평균 생존률** (스칼라, `[0,1]`) |

### X — 5비트 LUT

```
X = [DD, RSI, MA, BB, FX]   각 0 또는 1
예) DD+MA+FX → [1, 0, 1, 0, 1]
탐색공간 = 2⁵ − 1 = 31가지 (빈 조합 제외)
```

### Y — 스칼라, maximize

```
Y = mean over gyms ( survived_count / trials )
```

- `evolve.py`가 fitness 내림차순 정렬 → 상위 선발 → **최대화** 방향.
- 한 번의 전투 결과 `survived`(bool)는 **Y의 raw 재료**일 뿐, 목적함수가 아니다.
  랜덤 보정(±20) 때문에 한 판은 못 믿으므로 `trials`번 평균내 안정적 비율로 만든다.

```
survived(bool) ──trials 평균──▶ 체육관별 생존률 ──gym 평균──▶ Y(fitness)
   ① raw 관측                     ② 0~1                       ③ 목적함수
```

---

## 2. GA 4단계 (`evolve.py`)

| 단계 | 함수 | 하는 일 |
|------|------|---------|
| 평가 | `evaluate` | X를 trials번 도전시켜 Y(+체육관별 생존률) 계산 |
| 선택 | `select` | Y 높은 순 상위 절반을 부모로 (절단 선택) |
| 교배 | `crossover` | 두 부모의 5비트를 균등 교배 |
| 돌연변이 | `mutate` | 낮은 확률로 비트 하나 토글 (다양성 유지) |

`evolve(... on_generation=콜백)` — 세대마다 호출되는 콜백 훅(로깅/시각화/early stop 자리).

---

## 3. 현재의 한계 (의도된 것)

- 점수 체계가 "유전자 많을수록 무조건 유리" → GA가 **전 유전자 조합으로 수렴**한다.
- 탐색공간 31가지는 사실 완전탐색도 가능 → 여기서 GA는 **메커니즘 검증용 교보재**.
- 그래도 가치 있는 이유:
  1. GA 기계(선택→교배→돌연변이→세대)가 제대로 도는지 눈으로 검증.
  2. **최적 전략조차 어느 시장(체육관)에서 박살나는지** 관찰 → 다음 단계의 근거.

```
최종 챔피언(전 유전자)도:
  FINANCIAL_CRISIS  16.7%   ← 박살
  DOTCOM            43.3%
  RATE_SHOCK        90.0%
  COVID            100.0%
```

→ "왜 한 전략으로 모든 국면을 못 이기나?"가 보인다. 이게 다음 항목의 출발점.

## 4. 로드맵 — X축/Y축으로 확장

| | 지금 (v0.2) | 다음: 가중치 | 그다음: 다목적 |
|---|------------|-------------|---------------|
| **X** | on/off `{0,1}⁵` | 연속 가중치 `[0,1]⁵` | 가중치 + 예산제약 |
| **Y** | 스칼라 (평균 생존률) | 스칼라 | **벡터** (체육관별 생존률) |
| 탐색공간 | 31 (이산) | 무한 (연속) | 무한 (연속) |
| GA 역할 | 장식(완전탐색 가능) | **필수** | **필수** |

### X 확장 — 가중치(연속)

```
final_score = Σ wᵢ · signalᵢ      (지금은 wᵢ ∈ {0,1} 고정)
            → wᵢ ∈ [0,1] 연속으로 풀면 진짜 파라미터 최적화
```

⚠️ **함정:** 연속으로 풀어도 제약이 없으면 "모든 wᵢ 최대"로 또 수렴한다.
의미 있는 최적화가 되려면 둘 중 하나 필요:
1. **예산 제약** `Σ wᵢ = 1` (한정 자본 배분 = 포트폴리오)
2. **국면별 부호 반전** (어떤 시그널이 A국면엔 +, B국면엔 −)

### Y 확장 — 다목적

```
평균으로 뭉개면 "한 국면에서 처참히 죽는데 평균은 그럭저럭"인 전략이 숨는다.
→ 체육관별 생존률을 각각의 목적으로:
     Y = [ f_DOTCOM, f_FINCRISIS, f_COVID, f_RATESHOCK ]   (4목적)
→ 결과 = 최강 전략 1개가 아니라 전략 Pareto front
       = 국면별 특화 전략 + 올라운더 라인업
```

## 5. 구현 경로 — 손코딩 → 프레임워크

`evolve.py`(손코딩 GA)는 **원리 이해용**이다. 연속·다목적으로 가면
직접 짜지 말고 검증된 프레임워크로 갈아끼우는 게 정석:

```python
import optuna

study = optuna.create_study(
    directions=["maximize"] * 4,                 # 체육관 4개 = 4목적
    sampler=optuna.samplers.NSGAIIISampler(),    # 다목적 진화 샘플러
)

def objective(trial):
    w = [trial.suggest_float(g, 0.0, 1.0) for g in ALL_GENES]  # X = 연속 가중치
    return per_gym_survival(w)                                  # Y = 체육관별 벡터

study.optimize(objective, n_trials=2000, callbacks=[...])       # 콜백 훅
# study.best_trials  →  Pareto front
```

손코딩으로 X/Y/선택/교배의 감을 잡았으니, 위 교체는 작다.

---

## 6. 왜 토이인가

작은 31짜리 이산 공간에서 GA 메커니즘을 **눈으로** 확인한 뒤,
같은 구조(maximize Y over X)를 연속·다목적·대규모 공간으로 **그대로 확대**한다.
구조는 같고 스케일만 커진다 — 토이에서 익힌 직관이 1:1로 옮겨간다.
